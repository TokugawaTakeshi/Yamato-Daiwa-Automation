Terminology

YDA

YDA is the abbreviation of Yamato Daiwa Automation, the name of this tool about documentation which. Here, the Yamato Daiwa is the name of of Japanese startup, currently specializing on custom web development, and also on development of the librarires, frameworks and other tools (including YDA).

Project

In the programming context, the project — is the files and folders set which has the common parent directory and on which base the software product (web application, console application, library etc.) will be built. Why this term important is the YDA intended exactly for the working with projects, not with individual files.

Project build(ing)

In the "project building", the gerund "building" refers mainly on the process, namely the forming of the output files based on the source code and other source files. This process is the main thing which YDA doing.

The "build" could be the noun or verb.

  • As noun, the "build" in the project build means all output files of the project which usually has the common directory called "build directory". YDA does not require the single build directory, because is some cases it could be the obstacle for the solving of target problems.
  • As verb, the "build" in the build project means the various processes forming the output files such as source code compiling, images optimization etc.

Not all technologies requires the project building. For example, if the web site or web application has been written by pure HTML, CSS and PHP (the last one is interpreted programming language what means that source code files are being executed directly without compilation to the output executable files), in this case it is nothing to build. About YDA, it has being developed for the working with Pug, Stylus и TypeScript languages each of them intended to be transpiled to the output code (HTML, CSS and JavaScript respectively). Besides, output images files could differ with source ones because of automated optimization. In the case, when the files are not required any processing, the plain copying could be setup to not mix the source files and output ones.

Scenario

In the project building context, the task is the specialized work for the computer among project building automation. Currently the following primary tasks on which YDA is specializing could be highlighted.

  • The processing of the markup with source code on Pug, the preprocessor language.
  • The processing of the styles with source code on Stylus, the preprocessor language.
  • The processing of ECMAScript logic with source code on TypeScript (although it rarely called by "preprocessor language", it will be so until it's runtimes will become popular). The transpiling to both browser JavaScript and Node.js is available.
  • Images processing
  • The copying of fronts, videos, audios and ony other files

Task

In the context of YDA, and also Gulp, and similar project bulling tools, the scenario are the tasks been arranged into sequences (series) and parallels.

The correct tasks arranging is critical for both correct executing and performance optimization. For example, the markup processing is been executed only when files all other files types are ready to correctly resolve the paths to styles, images etc., and browser is opening automatically last of all, when the markup is ready.

Because the YDA is more high-level tool than Gulp, the users does not need to care about correct arranging of the tasks to series and parallels. It is enough to know which scenarios are available.

Currently, YDA supports 2 scenarios:

Project incremental building
First, the full project building is being executed, then on changing of the source files, the selective rebuilding will be. So, the application will be executed until will not be manually stopped or the critical error occurs. If required, the automatic browser opening (if the project has the client part) and/or automatic starting of the local server (if the project has the Node.js server part) could be setup.
Production-like project building
The project is being built, then the application will stop. The functionality like automatic reloading of the browser page is unavailable because of the scenario purpose.

Project Building Mode

Both scenarios could be executed in the multiple (project building) modes, each of them has specific feature. YDA is suggesting 5 following project building modes. They also could be considered as the stages, but not the stages of development of the software as a whole — the are the stages of the development of a separate page or a specific functionality.

Incremental

This adverb means that on such modes first full project building is being executed, а the with the changes of the source code, automatic selective rebuilding starts.

Static Preview
Designed and the development stage on which the engineer focuses on the markup and styles, while the JavaScript dynamics and server part (backend) is the matter of next stages. All problems related with the design (appearance) and adaptation to screens of various sized should be solved on this stage. In this mode, YDA case build the project such as the customers or managers could check the interim result without local server and any other specialized software.
Local Development
On this stage, unlike previous one, the full-fledged implementation of JavaScript dynamics and server part (backend) is being executed. On this stage, the project building tool must provide the optimized project rebuilding, automated reloading of the browser page, automated restarting of the local server with the changing of the files and other measured gives the comfortable software development with high-quality feedback including early notification about various problems such as the HTML validity violation.
Production-like

It is usual before the publishing of the web site or the web application they being launched on the test server with the limited access to make testers able to check the bugs which has not been detected on previous development stages. On the stages of this group, there is no the automated project rebuilding or automated browser reloading on updating of files anymore because built web site or web application intended to be launched on the server, not on the local machine. Although such builds must be close to the final one (the production one, that is why the modes of this group is being called "production-like"), some behaviour (for example, the logging level) could be different dependent on the specific mode.

Testing
Unlike the local development mode, the testers, managers and other people related with target web site or web application has access to the instance built in this mode. As it obvious from the name, the target of the current stage is the testing for the detecting of the bugs.
Staging
In large projects, besides the testing build, there is the similar one called "staging". The difference are usually small: it tends to be the usage of the data close with the real one, not the data designed for testing purposes. Thanks to it, the management and customers feels like the using the final published instance of the web site or web application.
Production
The web site of web application built in this mode intended to be published for the target audience. By comparison with other production-like modes, here the maximal security settings are required. Also, on this stage, the developers are tending to minimize the output to the browser console.

Entry Points and Partials

The entry point is the name of design pattern, in the general case it is the file, function or class method from which the program execution is been started. In the context of the certain technology, it could have more specific meaning.

Markup

In markup case, the entry point is the file representing the complete HTML file. Basically, HTML file could not be split to multiple files, but the pre-processors like Pug allows to do it on the source code stage for example, for the reusing of the specific code in multiple files. Here parent files (which are the entry points compiled to complete HTML files) and partials (children context dependent files which are not representing the valid HTML document and could be used only as the part of entry points must be distinguished. Although originally the «partial» does not assume the vertical hierarchy, in the context of the programming and declarative languages (especially pre-processors), often is does.

Styles

In styles case, the "entry point" term is unclear if to consider only output СSS-files. Unlike the HTML documents, in relation to cascading style sheets, the "complete" adverb is not applicable, because the styles related with certain HTML document could be arbitrary distributed between CSS files and also exist outside of the CSS files.

But, in the case of usage of CSS preprocessors, the entry point is the file including the source code, from which the conversion to output CSS code beings. The CSS preprocessors also allows to organize the styles by partial files which is not being conversed to separate CSS files but could be included to the entry points (in the Stylus preprocessor case it is possible by @require и @import directives).

There are no rules how many CSS files must be provided per HTML page, thus the styles entry point not necessary must include all styles exclusively for corresponding HTML document. But the spitting of the styles to CSS files is the important performance optimization measure (also, the caching of CSS files should be respected).
